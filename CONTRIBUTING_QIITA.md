# Qiita部門 ガイドライン

Qiita技術記事の執筆における共通ルールを定義します。全エージェントはこのガイドラインを遵守してください。

## トーン＆マナー

- **結論ファースト:** 記事冒頭で「この記事で何がわかるか」を明示する
- **再現可能な手順:** コマンドや手順は読者がそのまま実行できるレベルで記述する
- **環境情報の明記:** 使用した言語・フレームワーク・OSのバージョンを冒頭に記載する
- **ですます調:** 丁寧語（ですます調）で統一する
- **過剰な装飾を避ける:** 太字・絵文字の多用を避け、構造と内容で勝負する
- **読者目線:** 専門用語は初出時に簡潔に説明する。ただし対象読者のレベルに応じて調整する

## 構造

- **見出し階層:** H2 → H3 → H4 の順で使用。H1は使用しない（Qiitaではタイトルが H1 扱い）
- **導入セクション:** 記事冒頭に「この記事の対象読者」「前提知識」「ゴール」を記載する
- **まとめセクション:** 記事末尾に要点を箇条書きでまとめる
- **セクション間の接続:** 各セクションの冒頭で前セクションとの繋がりを簡潔に示す
- **目次の設計:** 見出しがそのまま右サイドバーの目次になるため、見出しだけで記事の流れがわかるように設計する

## フォーマット

### 記事ファイルの作成

- **記事ファイルは必ずプロジェクトルートで `npx qiita new` を実行して作成する**
- `public/` ディレクトリに手動でファイルを作成してはならない（Qiita CLIが管理するファイル名形式と整合しなくなるため）

### フロントマター

`npx qiita new` で生成されたファイルのフロントマターを以下の形式に編集する。

```yaml
---
title: '記事タイトル'
tags:
  - 'タグ名1'
  - 'タグ名2'
private: true
updated_at: ''
id: null
organization_url_name: null
slide: false
ignorePublish: false
---
```

- タグは最大5つ。Qiitaに既存のタグを優先的に使用する
- 初稿は必ず `private: true` で作成する

### Markdown記法

- **コードブロック:** 言語指定とファイル名を必ず記載する
  ````markdown
  ```typescript:src/example.ts
  const greeting: string = "Hello, Qiita!";
  ```
  ````
- **ノート記法:** 補足・警告に使用する
  ```markdown
  :::note info
  補足情報
  :::

  :::note warn
  注意事項
  :::

  :::note alert
  重要な警告
  :::
  ```
- **折りたたみ:** 長いコードや補足情報に使用する
  ```markdown
  <details><summary>クリックで展開</summary>

  折りたたまれたコンテンツ
  </details>
  ```
- **数式:** LaTeX記法（`$inline$` / `$$block$$`）
- **リンク:** 外部リンクには参照先の信頼性を確認し、公式ドキュメントを優先する

## 品質基準

- **技術的正確性:** コード・コマンドは実際に動作することを前提に記述する。未検証の場合はその旨を明記する
- **再現性:** 読者が記事の手順に従って同じ結果を得られること
- **実用性:** 実務やプロジェクトで活用できる内容であること
- **文字数:** 本文2,000〜6,000文字を目安とする（コードブロック除く）
- **コードブロック:** 最低1つ以上のコードブロックを含む。コピペで動作するコードを心がける
- **誤字脱字ゼロ:** レビューフェーズで必ずチェックする
- **リンク切れゼロ:** 参照URLの有効性を確認する

## タグ戦略

- トレンドタグや検索ボリュームの高いタグを優先的に使用する
- 技術名は正式名称のタグを使用する（例: `React` ではなく `react`、`TypeScript` ではなく `TypeScript`）
- タグは企画フェーズで選定し、レビューフェーズで最終確認する
- 最大5つのタグを有効活用する

## 禁止事項

- 他の記事やドキュメントのコピペ（引用は出典を明記すれば可）
- 古いバージョンの情報を最新として記述すること
- 検証していないコードを「動作確認済み」として掲載すること
- 特定の製品・サービスへの過度な宣伝
- 読者を誤解させる表現やタイトル詐欺
- H1見出し（`#`）の使用
