Claude Codeの出力品質は「プロンプトの書き方」だけでは決まりません。**AIが参照する情報環境全体を設計する「コンテキストエンジニアリング」**が、成果を左右する本質的なスキルです。

この記事では、Claude Codeに特化した5つのコンテキスト設計パターンを、Before/After形式で実践的に解説します。

## この記事の対象読者・前提知識・ゴール

- **対象読者:** Claude Codeを使い始めたが「出力の品質にムラがある」「長いセッションで精度が落ちる」と感じている中級エンジニア
- **前提知識:** Claude Codeの基本操作（プロンプト入力・ファイル編集・コマンド実行）ができること
- **ゴール:** コンテキストエンジニアリングの5つの設計パターンを理解し、Claude Codeの出力品質を安定させる方法を身につける
- **環境:** Claude Code 最新版（2026年4月時点）

## なぜ「プロンプト」だけでは限界なのか

プロンプトエンジニアリングは「何を聞くか」の技術です。一方、コンテキストエンジニアリングは**「何を渡し、何を渡さないか」を設計する技術**です。Claude Codeのようなエージェント型AIでは、プロンプトだけでなく、AIが参照する情報環境全体が出力品質を決定します。

| 観点 | プロンプトエンジニアリング | コンテキストエンジニアリング |
|:--|:--|:--|
| 対象 | 1回の指示文 | AIが参照する情報環境全体 |
| スコープ | 「何を聞くか」 | 「何を渡し、何を渡さないか」 |
| 持続性 | 会話ごとに消える | セッションを超えて永続化可能 |
| 設計対象 | テキスト | CLAUDE.md + MCP + Memory + プロンプト |
| 核心 | 良い質問の技術 | 有限な注意力の資源管理 |

Claude Code公式ドキュメントでも「**コンテキストウィンドウは管理する最も重要なリソースです**」と明記されています。コンテキストが肥大化すれば、Claudeは以前の指示を「忘れる」か、精度が低下します。

ここからは、この課題を解決する5つの設計パターンを順に解説します。

## パターン1 — CLAUDE.mdで「プロジェクトの常識」を永続化する

CLAUDE.mdは、Claude Codeがすべてのセッション開始時に自動で読み込む特別なファイルです。ここに「プロジェクトの常識」を記述することで、毎回の説明コストを省き、一貫した出力を実現できます。

### 書くべきもの・書くべきでないもの

CLAUDE.mdの品質は「何を書くか」ではなく**「何を書かないか」**で決まります。

| 書くべきもの | 書くべきでないもの |
|:--|:--|
| Claudeがコードから推測できないBashコマンド | Claudeがコードを読めば理解できること |
| デフォルトと異なるコードスタイルルール | 標準的な言語規約（PEP 8等） |
| テスト実行コマンドと推奨テストランナー | 詳細なAPIドキュメント |
| リポジトリのブランチ命名・PR規約 | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャ決定 | ファイルごとのコードベース説明 |

### 効果的なCLAUDE.mdの実例

```markdown:CLAUDE.md
# Code style
- Use ES modules (import/export), not CommonJS (require)
- Destructure imports when possible (e.g., import { foo } from 'bar')

# Workflow
- Run `npm run typecheck` after making code changes
- Run single test files, not the full suite (e.g., `npm test -- path/to/test`)

# Architecture
- API routes are in src/routes/, business logic in src/services/
- All DB access goes through src/db/repository.ts — no direct queries elsewhere
```

:::note warn
CLAUDE.mdが100行を超えると、ルールがノイズに埋もれてClaudeに無視されやすくなります。各行について「これを削除するとClaudeが間違いを犯すか？」を問い、不要な記述は削除してください。
:::

CLAUDE.mdは配置場所によってスコープが変わります。`~/.claude/CLAUDE.md`は全セッションに、プロジェクトルートの`CLAUDE.md`はそのプロジェクトに、子ディレクトリの`CLAUDE.md`はそのディレクトリ内のファイル操作時にオンデマンドで適用されます。モノレポでは親ディレクトリと子ディレクトリの階層構造を活用できます。

プロジェクトの常識を設定したら、次は外部ツールとの接続を設計します。

## パターン2 — MCPツールで「外の世界」を接続する

MCP（Model Context Protocol）は、Claude Codeが外部サービス（DB・API・監視ツール等）にアクセスするための標準インターフェースです。MCPサーバーを接続すると、Claudeはデータベースへのクエリ、Issue管理、デザインツール連携などを会話の中で実行できます。

### 設定例

プロジェクトルートの`.mcp.json`にMCPサーバーを定義します。

```json:.mcp.json
{
  "mcpServers": {
    "notion": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/claude-code-mcp-notion"],
      "env": {
        "NOTION_API_KEY": "secret_xxx"
      }
    }
  }
}
```

`claude mcp add`コマンドでも追加できます。プロジェクトスコープ（`.mcp.json`）とユーザースコープ（`~/.claude.json`）を使い分けましょう。

### よくある失敗パターンと対処法

| 失敗パターン | 症状 | 対処法 |
|:--|:--|:--|
| MCPサーバーの入れすぎ | ツール定義だけでコンテキストが肥大化 | 本当に必要なサーバーだけに厳選する |
| 認証情報のハードコード | チームで共有できない、漏洩リスク | 環境変数で管理し`.env`で注入する |
| 全機能を有効化 | 不要なツールがClaudeの判断を混乱させる | 使うツールだけを明示的に設定する |

:::note info
MCPサーバーの詳しい自作方法は「[実務で使えるMCPサーバー入門 — FastMCP 3.xでSlack通知ツールを30分で作る](https://qiita.com/NAKAJI_Rw/items/47cd7157ceffd4e49998)」で解説しています。
:::

外部ツールを接続したら、次はセッションをまたいだ記憶の管理を設計します。

## パターン3 — Memory & Skillsで「学習する文脈」を構築する

CLAUDE.mdは「プロジェクト全体の常識」を扱いますが、**会話の中で学んだこと**や**特定ドメインの手順書**はどこに置くべきでしょうか。Auto MemoryとSkillsが、この課題を解決します。

### Auto Memory — セッション間の学習

Auto Memoryは、Claudeが会話の中で得た知見をファイルに保存し、次のセッションで自動的に参照する仕組みです。ユーザーの好み、プロジェクトの方針、フィードバックなどがセッションを超えて蓄積されます。

### Skills — ドメイン知識のオンデマンド提供

Skillsは`.claude/skills/`にドメイン知識を定義するファイルです。CLAUDE.mdと異なり、**関連するときだけ自動的にロードされる**ため、コンテキストを常時消費しません。

```markdown:.claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints
- Version APIs in the URL path (/v1/, /v2/)
```

`/api-conventions`でいつでも呼び出せます。`disable-model-invocation: true`を設定すれば、手動トリガー専用にもできます。

### CLAUDE.md・Memory・Skillsの使い分け

| 機能 | 用途 | 読込タイミング | 適する情報 |
|:--|:--|:--|:--|
| CLAUDE.md | プロジェクトの常識 | 毎セッション自動 | ビルドコマンド、コード規約、アーキテクチャ方針 |
| Auto Memory | 学習した知見 | 毎セッション自動 | ユーザーの好み、プロジェクト状況、フィードバック |
| Skills | ドメイン知識・手順書 | オンデマンド | API設計規約、デプロイ手順、レビューチェックリスト |

**判断基準:** 毎回必要なら`CLAUDE.md`、Claudeが自動で学ぶべきなら`Memory`、特定タスク時だけ必要なら`Skills`に配置します。

永続的な文脈を設計したら、次は1回1回のプロンプトの品質を高めます。

## パターン4 — プロンプトに「検証基準」を埋め込む

Claude Codeは自律的にファイルを読み取り、コマンドを実行し、コードを書くエージェントです。しかし、**何をもって「成功」とするかを伝えなければ、正しく見えるが実際には動かないコードを生成する**可能性があります。

プロンプトに検証基準を埋め込むことで、Claudeが自己検証ループを回し、品質が安定します。

### Before/After比較

| 観点 | Before（曖昧なプロンプト） | After（検証基準付きプロンプト） |
|:--|:--|:--|
| 実装指示 | 「メールアドレスを検証する関数を作って」 | 「validateEmail関数を書いて。user@example.comはtrue、invalidはfalse、user@.comはfalseになること。実装後にテストを実行して」 |
| UI変更 | 「ダッシュボードをもっと良くして」 | 「[スクリーンショット]このデザインを実装して。結果のスクリーンショットを撮って元と比較して。違いをリストアップして修正して」 |
| バグ修正 | 「ログインバグを直して」 | 「セッションタイムアウト後にログインが失敗する。src/auth/のトークン更新を確認して。問題を再現する失敗テストを書いてから修正して」 |
| 調査 | 「このコードを調べて」 | 「サブエージェントを使って認証システムのトークン更新の仕組みと、既存のOAuthユーティリティを調査して」 |

### 検証基準付きプロンプトのテンプレート

```text
## タスク
{やりたいことを具体的に記述}

## 対象ファイル
{変更対象のファイルパスを明示}

## 成功条件
- {条件1: テストが通ること}
- {条件2: 既存のテストが壊れないこと}
- {条件3: 型チェックが通ること}

## 検証方法
{完了後に実行すべきコマンドやチェック手順}
```

:::note info
探索→計画→実装→検証の4フェーズに分けると、さらに精度が上がります。Plan Mode（`Ctrl+G`で切替）を活用して、実装前にClaudeにアプローチを計画させましょう。
:::

ここまでの4パターンで文脈を設計したら、最後にそれを維持する管理戦略を学びます。

## パターン5 — コンテキストウィンドウを「資源」として管理する

コンテキストウィンドウは有限資源です。会話が長くなるほど、ファイル読み取りやコマンド出力が蓄積し、Claudeのパフォーマンスが低下します。この資源を意識的に管理することが、長時間セッションでの品質維持に不可欠です。

### コンテキスト肥大化の症状と対策

| 症状 | 原因 | 対策 |
|:--|:--|:--|
| 以前の指示を「忘れる」 | コンテキスト満杯で古い情報が圧縮される | `/clear`でリセットして新しいプロンプトで再開する |
| 同じ間違いを何度も繰り返す | 失敗した試行がコンテキストにノイズとして残る | 2回修正して直らなければ`/clear`して再出発する |
| 無関係なファイルを読み始める | 関連しないタスクの文脈が混在している | タスクが変わるたびに`/clear`する |
| 応答が遅くなる | コンテキストが大きすぎる | `/compact`で会話を圧縮する |

### 3つの管理手法

**1. `/clear` — コンテキストの完全リセット**

タスクが変わるタイミングで`/clear`を実行し、コンテキストをゼロに戻します。学んだことは新しいプロンプトに組み込めば失われません。

**2. `/compact` — 選択的な圧縮**

`/compact`を実行すると、Claudeが会話の重要な部分（コードパターン、ファイル状態、主要な決定）を要約し、不要な部分を圧縮します。`/compact API変更に集中して`のようにフォーカスを指定することもできます。

**3. サブエージェント — コンテキストの分離**

調査や分析をサブエージェントに委譲すると、メインの会話コンテキストを汚さずに済みます。サブエージェントは別のコンテキストウィンドウで実行され、要約だけを返します。

```text
サブエージェントを使って、認証システムのトークン更新の仕組みと
既存のOAuthユーティリティがあるかを調査して。
```

:::note warn
**避けるべきパターン:** 1つのセッションで無関係なタスクを次々に処理する「キッチンシンクセッション」は、コンテキストが関係ない情報で埋まり、品質が急落します。タスクが変わったら`/clear`を習慣にしましょう。
:::

## 5つのパターンの全体像

ここまで解説した5つのパターンを整理します。

| パターン | 対象 | 永続性 | 主な効果 |
|:--|:--|:--|:--|
| 1. CLAUDE.md | プロジェクトの常識 | 永続（ファイル） | 毎回の説明コスト削減、一貫性の確保 |
| 2. MCPツール | 外部サービス接続 | 永続（設定ファイル） | 外部データへのアクセス、ワークフロー自動化 |
| 3. Memory & Skills | 学習・ドメイン知識 | 永続（ファイル） | セッション間の学習、オンデマンド知識提供 |
| 4. 検証基準付きプロンプト | 1回のタスク指示 | 一時的 | 自己検証ループによる品質安定 |
| 5. コンテキスト管理 | セッション全体 | セッション内 | 長時間セッションのパフォーマンス維持 |

**段階的に取り組む推奨順序:** まずCLAUDE.mdの整備（パターン1）とプロンプト改善（パターン4）から始め、効果を実感してからMCP（パターン2）、Memory & Skills（パターン3）、コンテキスト管理（パターン5）へ拡張していくのがおすすめです。

コンテキストエンジニアリングの核心は、**情報を増やすことではなく「何を入れないか」を決めること**です。Claude Code公式ドキュメントでも、CLAUDE.mdが長すぎるとルールが無視されると警告しています。1,300本の論文分析でも、情報の入れすぎがAI失敗の最大要因であることが示されています。

## まとめ

- **コンテキストエンジニアリング**は、プロンプトの書き方を超えた「AIの情報環境全体の設計」
- **CLAUDE.md**はプロジェクトの常識を永続化する。簡潔に保ち、100行を超えないこと
- **MCPツール**は外部サービスを接続する。必要なサーバーだけに厳選すること
- **Memory & Skills**でClaude Codeは「使うほど賢くなる」環境になる
- **検証基準付きプロンプト**でClaude Codeの自己検証ループを活用する
- **コンテキストウィンドウは有限資源**。`/clear`・`/compact`・サブエージェントで管理する
- 核心は「情報を増やすこと」ではなく**「何を入れないか」を決めること**
