## はじめに

AWSアカウントを開設して、最初にログインした状態のまま開発や設定を進めていませんか？
もしそのアカウントが「ルートユーザー（メールアドレスでログインするユーザー）」であるなら、少し立ち止まる必要があります。公式のベストプラクティスでも、日常的な作業においてルートユーザーを使用することは強く非推奨とされています。

この記事では、AWSを触り始めたばかりの方に向けて、なぜ「IAMユーザー」が必要なのかという概念から、安全な権限管理のベストプラクティス、そしてAWS CLIを用いた具体的な構築手順までを丁寧に解説します。

### この記事で得られる知識
- ルートユーザーとIAMユーザーの明確な違い
- 権限管理を安全に行うためのベストプラクティス
- AWS CLIを使ったIAMユーザーとグループの作成手順

### 対象読者
- AWSアカウントを作成したばかりの方
- セキュリティを意識した権限管理の基本を学びたいエンジニア
- マネジメントコンソールだけでなく、CLIでの操作にも慣れていきたい方


## 前提条件/環境

本記事のハンズオン（実践編）では、以下の環境を前提としています。

- AWSアカウントが作成済みであること
- ご自身のローカル環境に **AWS CLI** がインストールされ、管理者権限を持つプロファイルで初期設定（`aws configure`）が完了していること


## そもそもAWS IAMユーザーとは？

IAM（Identity and Access Management：アイアム）は、AWSリソースへのアクセスを安全に管理するためのサービスです。その中でも「IAMユーザー」は最も基本となる概念です。

### ルートユーザーとIAMユーザーの違い
この2つの違いは、「**家のマスターキー**」と「**特定の部屋だけ開く合鍵**」に例えると分かりやすいです。

- **ルートユーザー（マスターキー）**: AWSアカウントを作成した際のアカウントです。AWSの解約を含め、すべてのサービスに対する完全なアクセス権を持ちます。万が一この情報が漏洩すると、アカウントを乗っ取られ、甚大な被害（不正な仮想通貨マイニングによる高額請求など）を受ける可能性があります
- **IAMユーザー（合鍵）**: 日常の作業を行うために作成するユーザーです。「EC2サーバーを立てる権限だけ」「S3のデータを読み取る権限だけ」といったように、必要最小限の権限（合鍵）を渡すことができます

「**ルートユーザーは厳重に金庫にしまい、普段はIAMユーザーという合鍵を作って作業する**」というのが、AWSにおける鉄則です。

### IAMの3つの基本要素
IAMを理解する上で、以下の3つの関係性を覚えておきましょう。

1. **ユーザー (User)**: AWSを操作する「人」や「アプリケーション」そのものです
2. **ポリシー (Policy)**: 「どのAWSサービスに対して、何ができるか（またはできないか）」を定義したルールブック（JSON形式のドキュメント）です
3. **グループ (Group)**: 複数のユーザーを束ねる箱です

## 安全に管理するための3つのベストプラクティス

ただIAMユーザーを作れば安全、というわけではありません。以下の3つの原則を守ることが重要です。

### 1. 最小権限の原則（Least Privilege）
ユーザーには、そのタスクを実行するために必要な権限のみを付与してください。「とりあえず面倒だから `AdministratorAccess`（全権限）を付与する」というのは非常に危険なアンチパターンです。

### 2. ポリシーは「グループ」にアタッチする
IAMユーザー個人に直接ポリシーを紐付ける（インラインポリシーなど）ことは避けましょう。
例えば「開発者」という**グループ**を作成し、そのグループに「EC2とS3を操作できるポリシー」を付与します。そして、新しい開発メンバーが入ってきたら、その人を「開発者グループ」に追加するだけで済みます。これにより、管理漏れを防ぎ、見通しを良くすることができます。

### 3. MFA（多要素認証）の有効化
マネジメントコンソールにログインするIAMユーザー（およびルートユーザー）には、必ずMFA（スマートフォンアプリ等を使ったワンタイムパスワード）を設定しましょう。パスワードの漏洩に対する強力な防波堤になります。

## 【実践】AWS CLIでIAMユーザーを作成してみよう

概念を理解したところで、実際に手を動かしてみましょう。
今回はマネジメントコンソール（画面操作）ではなく、実務でよく使われる **AWS CLI** を使って、ベストプラクティスに沿った構成を作成します。

**【今回のシナリオ】**
S3の読み取り権限だけを持つ「開発者グループ」を作成し、そこに新しい「IAMユーザー」を追加して、プログラムからアクセスするためのアクセスキーを発行します。

### ステップ1: IAMグループの作成
まずはユーザーを入れるための「箱」であるグループを作成します。

```bash
# 'Developers' という名前のIAMグループを作成します
aws iam create-group --group-name Developers
```

### ステップ2: グループへのポリシーのアタッチ
作成したグループに、AWSが事前に用意している管理ポリシー（S3の読み取り専用権限）を付与します。

```bash
# 'AmazonS3ReadOnlyAccess' ポリシーを 'Developers' グループにアタッチします
# ARN (Amazon Resource Name) は、AWSリソースを一意に特定するためのIDです
aws iam attach-group-policy \
    --group-name Developers \
    --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

### ステップ3: IAMユーザーの作成
次に、新しいIAMユーザーを作成します。

```bash
# 'dev-user' という名前のIAMユーザーを作成します
aws iam create-user --user-name dev-user
```

### ステップ4: ユーザーをグループに追加
作成したユーザーを、ステップ2で権限を付与したグループに所属させます。これで `dev-user` はS3の読み取りができるようになります。

```bash
# 'dev-user' を 'Developers' グループに追加します
aws iam add-user-to-group \
    --user-name dev-user \
    --group-name Developers
```

### ステップ5: アクセスキーの作成
最後に、このユーザーがCLIやプログラムからAWSを操作するための「アクセスキー」を発行します。

```bash
# 'dev-user' のアクセスキーとシークレットキーを作成します
aws iam create-access-key --user-name dev-user
```

**【⚠️超重要：実行結果に関する注意点】**
上記のコマンドを実行すると、以下のようなJSON形式の結果が返ってきます。

```json
{
    "AccessKey": {
        "UserName": "dev-user",
        "AccessKeyId": "AKIAIOSFODNN7EXAMPLE",
        "Status": "Active",
        "SecretAccessKey": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
        "CreateDate": "2023-10-01T12:00:00+00:00"
    }
}
```

ここで表示される `SecretAccessKey` は、**この画面でしか確認できません**。絶対にGitHubなどのパブリックなリポジトリにコミットしたり、第三者に教えたりしないでください。漏洩した場合、あなたのアカウントが不正利用される原因となります。

## まとめ

この記事では、AWSアカウント作成直後に必ず意識すべき「IAM」の基本と、ベストプラクティスに沿った構築方法を解説しました。

- ルートユーザーでの日常作業は避け、必ずIAMを利用する
- 権限は最小限にし、ユーザー個人ではなくグループに対して付与する
- パスワードだけでなくMFA（多要素認証）を有効化する
- アクセスキーは絶対外部に漏らさない

**【次のステップへ】**
最近のAWS環境では、人間のユーザー（従業員など）がコンソールにログインする場合、個別のIAMユーザーを作成するのではなく、よりスケーラブルでセキュアな **「AWS IAM Identity Center (旧 AWS SSO)」** の利用が推奨されています。
しかし、概念の基盤となる「ポリシー」や、プログラム用のアクセスキーを発行する仕組みとしては、依然として今回学んだIAMの知識が必須となります。
