# Claude Code 部署設立仕様書

> この仕様書をClaude Codeに渡し、対話形式で必要事項を回答することで、新しい部署（Agent Teams）プロジェクトを自動生成できます。

---

## 概要

本仕様書は、Zenn部門（編集部）をリファレンス実装として、同様のAIエージェントチーム（部署）を新規設立するための手順・構成・テンプレートを定義します。

Claude Codeに本仕様書を渡し、以下のプロンプトで実行してください:

```
この仕様書に従って新しい部署を設立してください。対話形式で必要な情報をヒアリングし、プロジェクトを構築してください。
```

---

## 1. ヒアリング項目

部署設立にあたり、以下の情報をユーザーから収集する。不明な項目は提案・補完してよい。

### 1.1 基本情報

| 項目 | 説明 | 例（Zenn部門の場合） |
|------|------|---------------------|
| **部署名** | 組織上の部署名称 | 編集部 |
| **部門名** | 部署内の部門名称 | Zenn部門 |
| **プロジェクト名** | ディレクトリ名（英字） | Zenn |
| **ミッション** | この部門が達成すべき目標 | Zenn技術記事の企画・執筆・レビュー・公開 |
| **最終成果物** | 部門が生成する最終的なアウトプット | Zenn記事（Markdown） |
| **配置パス** | プロジェクトの配置先 | `C:\Workspace\ClaudeCodeCompany\{部署名}\{プロジェクト名}` |

### 1.2 エージェント設計

| 項目 | 説明 | 例 |
|------|------|-----|
| **エージェント一覧** | 各AIの名前・役割・担当フェーズ | リサーチャーAI、ライターAI、レビュアーAI... |
| **ワークフロー** | フェーズの実行順序と依存関係 | 企画→執筆→レビュー→公開 |
| **人間の介入ポイント** | ユーザー確認・承認が必要なタイミング | 各フェーズ完了後 |
| **統括エージェント** | 全フェーズを一括実行するオーケストレーター | `/zenn-full` 相当 |

### 1.3 外部連携（オプション）

| 項目 | 説明 | 例 |
|------|------|-----|
| **外部サービス** | Notion, GA, Slack等 | Notion同期, Google Analytics |
| **デプロイ方法** | 成果物の公開・配信方法 | git push → Zenn自動反映 |
| **CI/CD** | GitHub Actionsなどの自動化 | 週次アナリティクス取得 |

## 2. プロジェクト構造テンプレート

ヒアリング結果をもとに以下の構造を生成する。`{prefix}` は部門の短縮名（例: `zenn`）で置換する。

```
{プロジェクト名}/
├── .claude/
│   ├── settings.json          # ツール権限設定
│   ├── rules/                 # コンテキストルール（自動適用）
│   │   ├── {ルール1}.md
│   │   ├── {ルール2}.md
│   │   └── ...
│   └── skills/                # スラッシュコマンド（スキル）
│       ├── {prefix}-{役割1}/
│       │   └── SKILL.md
│       ├── {prefix}-{役割2}/
│       │   └── SKILL.md
│       ├── {prefix}-report/
│       │   └── SKILL.md
│       ├── {prefix}-knowledge/
│       │   └── SKILL.md
│       └── {prefix}-full/
│           └── SKILL.md
├── context/                   # ナレッジ蓄積
│   ├── report/                # 作業レポート（未処理）
│   ├── knowledge/             # 抽出済みナレッジ
│   ├── analytics/             # 分析データ（オプション）
│   └── old_report/            # 処理済みアーカイブ
├── {成果物ディレクトリ}/       # 部門固有の成果物格納先
├── scripts/                   # 自動化スクリプト（オプション）
├── prompt/                    # 設立経緯・プロンプト記録
│   └── establishment.md
├── CLAUDE.md                  # プロジェクト指示書（Claude Code用）
├── AGENTS.md                  # エージェント一覧
├── CONTRIBUTING_{PREFIX}.md   # 部門共通ガイドライン
├── README.md                  # プロジェクト概要
├── package.json               # スクリプト・依存関係（必要時）
├── .gitignore
└── .env.example               # 環境変数テンプレート（必要時）
```

## 3. ファイル生成仕様

### 3.1 `.claude/settings.json`

```json
{
  "permissions": {
    "allow": [
      "Bash(*)",
      "Edit(*)",
      "Write(*)",
      "Read(*)",
      "Glob(*)",
      "Grep(*)",
      "WebFetch(*)",
      "WebSearch(*)"
    ]
  }
}
```

### 3.2 SKILL.md（スキルファイル）

各エージェントのスキルファイルは以下の構造で作成する:

```markdown
---
name: {prefix}-{役割}
description: {エージェント名} — {一行の役割説明}
---

# {エージェント名} — {役割説明}

あなたは{部門の成果物}の{担当業務}を担当する{エージェント名}です。
{このエージェントが達成すべきことの概要}

## 実行手順

1. **{ステップ1名}:** {具体的な操作}
2. **{ステップ2名}:** {具体的な操作}
3. **{ステップ3名}:** {具体的な操作}
4. **保存:** `{保存先パス}` に保存する

## {業務固有のルール見出し}

- {ルール1}
- {ルール2}
- {ルール3}

## 参照先

- {参照元1}: `{パス}`
- {参照元2}: `{パス}`
- ナレッジ: `context/knowledge/`
- 過去レポート: `context/report/`

## 注意事項

- `context/report/` に過去のレポートがあれば参照し、過去の改善点を活かすこと
- 完了後、ユーザー（管理者）に確認・承認を求めること

$ARGUMENTS
```

### 3.3 ルールファイル（`.claude/rules/*.md`）

ルールファイルは特定のファイルパスに対して自動適用されるコンテキストルール。

```markdown
---
description: {このルールが何をするかの一行説明}
paths: ["{対象ファイルのglob}", "{対象ファイルのglob}"]
---

# {ルール名}

## 必須参照
- {ガイドラインファイル}
- {ナレッジファイル}

## {ルールカテゴリ}
- {具体的なルール}
- {具体的なルール}
```

### 3.4 統括エージェント（`{prefix}-full`）

全フェーズを順番に実行するオーケストレーター。以下の構造で作成する:

```markdown
---
name: {prefix}-full
description: {部門名}フルワークフロー — 全フェーズ一括実行
---

# {部門名}フルワークフロー — 全フェーズ一括実行

あなたは{部門の成果物}の全制作工程を一括で担当するAIチームの統括エージェントです。
以下のフェーズを順番に実行し、{開始}から{終了}までを一気通貫で行ってください。

各フェーズは対応するスラッシュコマンドを `Skill` ツールで実行してください。
各フェーズの完了後、ユーザー（管理者）に結果を提示し、確認・承認を得てから次のフェーズへ進んでください。

---

## フェーズ1: {フェーズ名}（{担当エージェント}）

`/{prefix}-{役割}` を実行する。

> **ユーザー確認ポイント:** {何を確認するか}

---

（以降、全フェーズを同様に記述）

---

## 最終フェーズ: デプロイ（git push）

全成果物をGitにコミットし、リモートリポジトリへpushする。

1. 成果物ファイルをステージングする
2. コミットメッセージの形式: `feat({カテゴリ}): {成果物タイトルの要約}`
3. リモートリポジトリへpushする

> **ユーザー確認ポイント:** push前にコミット内容を提示し、確認を得てからpushする

$ARGUMENTS
```

### 3.5 レポートAI（`{prefix}-report`）

作業プロセスをドキュメント化し、ナレッジとして蓄積する共通エージェント。

```markdown
---
name: {prefix}-report
description: レポートAI — 作業レポートの作成
---

# レポートAI — 作業レポートの作成

あなたは{部門の成果物}の作成プロセスをドキュメント化するレポートAIです。
成果物作成の工程・判断・修正履歴をまとめたレポートを作成してください。

## 実行手順

1. **成果物の確認:** 今回作成された成果物を確認する
2. **Gitログの確認:** 関連するコミット履歴を確認する
3. **過去レポートの参照:** `context/report/` の過去レポートを参照する
4. **レポート作成:** 以下の構造でレポートを作成する
5. **保存:** `context/report/report_YYYYMMDD_{テーマ名}.md` に保存する

## レポート構造

```
# 作業レポート: {テーマ名}

## 基本情報
- 作成日: YYYY-MM-DD
- テーマ: {テーマ}
- 成果物: {ファイルパス}

## フェーズ別サマリ
{各フェーズでの作業内容・所要判断}

## 判断・修正の経緯
{重要な判断ポイントと理由}

## レビュー指摘事項
{レビューで指摘された内容と対応}

## 改善点・学び
{次回に活かせるポイント}

## 統計情報
- 文字数: {N}文字
- セクション数: {N}
- コードブロック数: {N}
```

## 注意事項

- 過去レポートの傾向を分析し、繰り返し発生する課題があれば明記すること
- レポート作成後、外部連携（Notion同期など）が設定されている場合は実行すること

$ARGUMENTS
```

### 3.6 ナレッジAI（`{prefix}-knowledge`）

レポートと分析データからナレッジを抽出・更新する共通エージェント。

```markdown
---
name: {prefix}-knowledge
description: ナレッジAI — レポート＋分析データからナレッジを作成・更新
---

# ナレッジAI — ナレッジの作成・更新

あなたはレポートと分析データからナレッジを抽出し、`context/knowledge/` に蓄積するナレッジAIです。

## データソース

1. **レポート:** `context/report/` 内の未処理レポート
2. **分析データ:** `context/analytics/` 内のデータファイル（存在する場合）
3. **既存ナレッジ:** `context/knowledge/` 内のナレッジファイル
4. **既存成果物:** 成果物ディレクトリ内のファイル

## ナレッジファイル

以下のファイルを作成・更新する（部門に応じてカスタマイズ）:

- `context/knowledge/workflow-patterns.md` — 作業パターン・ベストプラクティス
- `context/knowledge/review-lessons.md` — レビューからの教訓
- `context/knowledge/analytics-insights.md` — 分析データからの知見
- `context/knowledge/production-tips.md` — 制作ノウハウ

## 実行手順

1. `context/report/` の未処理レポートを読み込む
2. `context/analytics/` のデータを読み込む（存在する場合）
3. 既存のナレッジファイルを読み込む
4. 新しい知見を抽出し、ナレッジファイルを更新する
5. 処理済みファイルを `context/old_report/YYYY-MM-DD/` へ移動する

## 注意事項

- 既存ナレッジを削除せず、追記・更新する
- 更新日時を各ファイルに記録する
- ナレッジの重複を避け、統合すること

$ARGUMENTS
```

## 4. 設立手順（Claude Codeが実行する）

### Step 1: ヒアリング

ユーザーに対して「1. ヒアリング項目」の各項目を質問する。
一度に全項目を聞くのではなく、以下の順序でグループ化して聞く:

1. **基本情報**: 部署名、部門名、ミッション、最終成果物
2. **エージェント設計**: 必要なエージェント、ワークフロー、人間の介入ポイント
3. **ガイドライン**: 成果物の品質基準、トーン＆マナー、禁止事項
4. **外部連携**（オプション）: 使用する外部サービス

各グループのヒアリング後、理解した内容をまとめてユーザーに確認を求める。

### Step 2: 設計確認

ヒアリング結果から以下を設計し、ユーザーに提示して承認を得る:

- ディレクトリ構成図
- エージェント一覧（名前・コマンド名・役割）
- ワークフロー図
- ルールファイル一覧

### Step 3: プロジェクト生成

承認後、以下の順序でファイルを生成する:

1. ディレクトリ構造の作成
2. `.claude/settings.json`
3. `CONTRIBUTING_{PREFIX}.md`（ガイドライン）
4. `.claude/rules/*.md`（ルールファイル）
5. `.claude/skills/*/SKILL.md`（各エージェントのスキルファイル）
6. `CLAUDE.md`（プロジェクト指示書）
7. `AGENTS.md`（エージェント一覧）
8. `README.md`（プロジェクト概要）
9. `prompt/establishment.md`（設立経緯の記録）
10. `.gitignore`
11. `.env.example`（必要な場合）
12. `package.json`（スクリプトが必要な場合）
13. `scripts/`（自動化スクリプトが必要な場合）

### Step 4: git初期化

```bash
cd {プロジェクトパス}
git init
git add .
git commit -m "feat: {部門名} Agent Teams 設立"
```

### Step 5: 動作確認

生成されたスキルの一覧を表示し、各コマンドが正しく認識されることを確認する。

## 5. 設計原則

新しい部署を設計する際、以下の原則を守ること:

### 5.1 エージェント設計

- **単一責任:** 各エージェントは1つの明確な役割を持つ
- **入出力の明確化:** 各エージェントの入力元と出力先を明確にする
- **人間の承認ゲート:** 重要なフェーズ間にはユーザー確認を挟む
- **ナレッジ循環:** レポート→ナレッジ→作業参照のフィードバックループを組み込む

### 5.2 共通エージェント

以下の3エージェントはどの部門でも必ず設置する:

| エージェント | コマンド | 必須理由 |
|---|---|---|
| **レポートAI** | `/{prefix}-report` | 作業プロセスの記録・ナレッジ蓄積の起点 |
| **ナレッジAI** | `/{prefix}-knowledge` | レポートからの知見抽出・品質の継続的改善 |
| **統括エージェント** | `/{prefix}-full` | 全フェーズの一括実行・ワークフロー管理 |

### 5.3 ナレッジ蓄積

- `context/report/` → 個別作業の記録
- `context/knowledge/` → レポートから抽出した汎用的な知見
- `context/old_report/` → 処理済みレポートのアーカイブ
- このサイクルにより、部門は作業を重ねるごとに品質が向上する

### 5.4 ファイル命名規則

| 種別 | 命名規則 | 例 |
|------|---------|-----|
| スキル名 | `{prefix}-{役割}` | `zenn-write`, `blog-edit` |
| レポート | `report_YYYYMMDD_{テーマ名}.md` | `report_20260407_typescript.md` |
| ルールファイル | `{カテゴリ}.md` | `writing.md`, `review.md` |
| ガイドライン | `CONTRIBUTING_{PREFIX}.md` | `CONTRIBUTING_ZENN.md` |
