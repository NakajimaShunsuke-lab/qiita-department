## 見出し構成案

### H2: この記事の対象読者・前提知識・ゴール
- 対象読者: Claude Codeを日常利用しているが、プロンプトの再入力・似た指示の繰り返しに疲れを感じる中級エンジニア
- 前提知識: Claude Codeの基本操作とCLAUDE.mdの書き方
- ゴール: Skillsの構造（SKILL.md + frontmatter）を理解し、5つの設計パターンを使い分けて自分/チームの定型業務をAIに移植できるようになる
- 環境情報: Claude Code 2.1.108以降、Claude Opus 4.6/4.7

### H2: なぜ今「Skills」なのか — プロンプトとの決定的な違い
- 結論ファースト: Skillsは「いつ・どの知識を・どう使うか」をファイルシステム上に記述する仕組み。プロンプト・CLAUDE.md・MCPとは役割が異なる
- 4要素の比較表（表1: CLAUDE.md / プロンプト / Skills / MCPの使い分け）
- Progressive Disclosure（段階的開示）の仕組みを3レベルで整理（表2: Level 1 メタデータ / Level 2 SKILL.md本文 / Level 3 付属ファイル）
- Claude Code 2.1.108の1時間キャッシュとの相性
- **接続文:** Skillsが効く場面が分かったら、次は最小構造を組み立てる

### H2: SKILL.mdの最小構造 — 5分で動くSkillを作る
- 結論ファースト: SKILL.mdはfrontmatter（name・description）+ 指示本文の2要素だけで動く。`description` の書き方が起動精度を決める
- ディレクトリ配置の2択（`.claude/skills/` vs `~/.claude/skills/`）を表で整理（表3: プロジェクト単位 vs グローバル単位）
- 最小SKILL.mdのコード例（frontmatter + 10行程度の指示）
- descriptionの書き方3原則（①具体的な動詞で開始 ②対象を明示 ③起動条件を1〜2文に収める）
- 公式仕様の制約（name 64文字・description 1024文字・予約語回避）
- **接続文:** 最小構造を理解したら、実務で使える5つの設計パターンを見ていく

### H2: パターン1 — コード生成ガード型（Guard Skill）
- 結論ファースト: コーディング規約や命名規則など、実装時に必ず適用したい制約を注入するパターン。AIの生成品質を底上げする
- ユースケース例: React命名規則の統一、SQLクエリのセキュリティチェック、コミットメッセージ規約
- SKILL.mdのコード例（命名規則ガードの実装、15行程度）
- 起動条件設計のコツ（「コードを書く」系の曖昧な語を避け、具体的なファイル種別・対象を記述する）
- アンチパターン: 長すぎる規約本文 → Level 2で5kトークン超過するとキャッシュ効率が落ちる
- **接続文:** 制約の注入パターンが分かったら、次は手順の漏れを防ぐパターン

### H2: パターン2 — チェックリスト型（Checklist Skill）
- 結論ファースト: PR前チェック・リリース手順・セキュリティ確認など、「漏れたら困る」定型手順をAIに委譲するパターン
- ユースケース例: PR作成前チェックリスト、依存関係アップデート前の確認、本番デプロイ前の事前確認
- SKILL.mdのコード例（PR前チェックリストSkill、20行程度）
- 承認ポイントの埋め込み方（取り消し困難な操作には必ず `ユーザー確認を挟む` の明文化）
- Before/After比較表（表4: 手動チェック vs Skillチェック — 所要時間・漏れ率・再現性）
- **接続文:** 定型手順を委譲できたら、次は情報収集の自動化

### H2: パターン3 — 情報収集型（Research Skill）
- 結論ファースト: Webからの情報収集・既存ドキュメント横断検索・トレンド調査など、複数ソースから情報を集めて要約するパターン
- ユースケース例: 週次トレンド調査、ライブラリ選定リサーチ、社内ドキュメント横断検索
- SKILL.mdのコード例（トレンド調査Skill、WebSearch/WebFetch指示含む、15行程度）
- Level 3の付属ファイル活用（参照すべきURL一覧・調査テンプレートをバンドルする）
- 注意点: 外部ネットワークアクセスはClaude Code環境のみ可。APIでは制約あり
- **接続文:** 情報収集を自動化できたら、次はフォーマット変換

### H2: パターン4 — フォーマット変換型（Transform Skill）
- 結論ファースト: 議事録・ログ・設計書など、入力フォーマットから出力フォーマットへの変換を規則ベースで実行するパターン。出力のブレを最小化する
- ユースケース例: 議事録→Notion形式変換、エラーログ→Slack通知用サマリ、プロトタイプコード→本番品質コード
- SKILL.mdのコード例（議事録→構造化MarkdownSkill、出力テンプレート埋め込み、20行程度）
- Level 3でテンプレートファイルを同梱する設計例（`templates/meeting-note.md`）
- 公式の推奨: 決定的な処理はスクリプトに逃がし、コンテキスト消費をゼロにする
- **接続文:** 単発変換パターンが分かったら、複数ステップを束ねる最終パターン

### H2: パターン5 — マルチステップ実行型（Pipeline Skill）
- 結論ファースト: 「調査→構成案→執筆→レビュー」のように、複数フェーズと承認ポイントを持つワークフロー全体を1つのSkillで記述するパターン
- ユースケース例: 記事執筆パイプライン、コードレビュー→PRコメント投稿、段階的デプロイ
- SKILL.mdのコード例（記事執筆パイプラインSkill、承認ポイント付き、25行程度）
- フェーズ設計のコツ（表5: 各フェーズの入力・出力・承認要否）
- サブエージェントや他Skillとの組み合わせ（Progressive Disclosureを活用してコンテキスト肥大化を防ぐ）
- アンチパターン: 全工程を1つのSkillに詰め込む → 500行超のSKILL.mdは保守不能。Level 3のファイル分割で整理する
- **接続文:** 5つのパターンが揃ったら、全体像と導入順序を整理する

### H2: 5つのパターンの全体像と導入順序
- 5パターンの対比表（表6: パターン名・典型ユースケース・起動頻度・Progressive Disclosure活用度）
- 段階的導入の推奨順序: Guard（最小リスク）→ Checklist → Transform → Research → Pipeline（最難）
- 「完璧さより動作を優先」「頻度の高い1タスクから始める」という公式推奨を引用
- チーム展開時のポイント（プロジェクト単位で`.claude/skills/`配下にコミットする運用）

## 差別化ポイント
1. **「設計パターン集」としての横断整理**: 既存記事（書き方完全ガイド・superpowers解説等）は「特定のSkill例の紹介」が中心。本記事は**ユースケース型5パターン**に抽象化し、読者が自分の業務にマッピングできる構造を提供する
2. **表6つ＋Before/After構成**: 自チームのGA実績（比較表6つでGA滞在90.4秒）を活かし、表主体の構成で深読みを促す
3. **Progressive Disclosure（3レベル）を正確に説明**: 公式ドキュメントに準拠した階層モデルを日本語で整理。誤解されがちな「Skillは全部読み込まれる」という誤認識を解く
4. **アンチパターン明記**: 各パターンに「失敗例」を含め、読者が実装時に踏む地雷を事前に示す
5. **自チームシリーズとの接続**: Agent Teams→Managed Agents→MCP→コンテキストエンジニアリング→FW比較→**本記事**、のシリーズ延長線として位置づけ、内部リンクで回遊を促す
6. **Claude Code 2.1.108（1時間キャッシュ）・Opus 4.7との組合せに言及**: 4月最新機能を実用視点でカバーし、旬を逃さない
